# Future Game-Loop Mechanics

## Purpose

This page describes how MonkeyWifHat could later introduce token-based participation loops without corrupting the clean launch tokenomics.

The goal is not to engineer a fake financial machine. The goal is to create optional, visible activities that have both a faucet and a sink.

## Design Rule

Any future game loop should be easy to explain in one sentence:

- users do a visible action
- users can earn tokens from that action under disclosed rules
- users can also spend tokens in optional ways tied to the same activity

If the loop needs jargon to justify itself, it is not ready.

The cleanest long-term model is to separate token use from reputation use:

- the token moves between wallets and into the game environment
- reputation stays attached to the player account, character, profile, or canonical series identity

That separation matters because reputation should measure contribution, consistency, and standing. It should not be directly purchasable in the same way the token is transferable.

## Faucet And Sink Model

For MonkeyWifHat, a healthy future loop should have both sides.

The token should function as the portable asset inside the game environment.

Reputation should function more like karma:

- earned through visible participation
- accumulated over time
- useful for status, unlock paths, and series recognition
- not freely tradable like the token itself

### Faucet

A faucet is how tokens re-enter active user hands through visible participation.

Acceptable faucet sources:

- contest rewards
- creator bounties
- campaign leaderboard payouts
- spotlight rewards
- disclosed partner-sponsored event rewards

Bad faucet sources:

- secret insider distributions
- vague ecosystem rewards with no criteria
- uncapped emissions that rewrite the fixed supply story

MonkeyWifHat should keep faucets budgeted, disclosed, and tied to actual activity.

Reputation faucets should be even stricter than token faucets.

Good reputation faucets:

- completing canonical challenges
- winning or placing in official series events
- consistent contribution over repeated cycles
- verified creative or gameplay contributions recognized by the system

Bad reputation faucets:

- buying reputation directly with tokens
- granting reputation for ordinary transfers
- opaque moderator handouts with no visible basis

### Sink

A sink is how some tokens leave active circulation because a user chose to spend them in a real activity.

Acceptable sink uses:

- event entry fees
- paid submissions
- cosmetic upgrades or vanity naming rights
- access to limited themed experiences
- optional challenge or tournament entry

The sink should come from optional actions, not from ordinary transfers.

Reputation should usually not be burned in the same way tokens are burned.

The cleaner approach is:

- tokens can be spent, retired, or burned in optional actions
- reputation can unlock or prioritize opportunities, but mainly grows or decays through use and inactivity rules

That makes reputation feel closer to karma or standing than currency.

## Game Environment Deployment Model

If MonkeyWifHat expands into a canonical game series, the token should enter the environment in a narrow, understandable way.

The intended nesting should be:

1. canonical series as the outer world
2. a named mode such as Hat Arena inside that world
3. token entry, faucet, sink, and reputation scoring inside the mode

That means the token math should serve the mode, and the mode should serve the series.

Recommended model:

- the token is used for entry, optional actions, themed upgrades, event access, or canon-linked choices
- the token can move into the game environment when a player chooses to use it
- once inside the loop, some token use may feed rewards, some may feed budgets, and a small portion may act as the sink
- the token should not automatically become reputation just because it was spent

This keeps the economic layer and the standing layer separate.

## Reputation And Canonical Series Model

The canonical series should use a separate reputation layer that behaves more like Reddit karma than like a tradable asset.

Recommended characteristics:

- reputation is earned, not bought outright
- reputation tracks visible history inside the series
- reputation can influence ranking, eligibility, canon placement, or access to special roles
- reputation should reset, decay, or seasonally re-balance only under disclosed rules
- reputation should not be freely transferable between players

That model lets the token be economically useful without making the narrative hierarchy pay-to-win.

In practical terms:

- tokens get you into the activity or let you express preference inside it
- reputation records what you actually did there
- canon status comes more from repeated contribution than from raw token balance

## Recommended Rule Of Balance

If a future loop exists, the project should prefer:

- a visible faucet that rewards participation
- a smaller visible sink that comes from optional spending

The default ratio should stay conservative.

Recommended starting rule:

- roughly `90%` of tokens spent in a loop should return to rewards, community pools, or disclosed program use
- no more than `10%` should be burned or permanently retired unless the reason is unusually strong and publicly explained

This creates light friction without turning the project into a token tax system.

## Example Loop Structure

Simple example:

- users join a biweekly challenge
- winners or featured contributors receive tokens from a disclosed faucet budget
- users can also spend tokens for optional premium submissions, vanity placements, or themed access
- most spent tokens return to event pools or visible budgets
- a small portion is burned or retired as the sink
- separate reputation points track placements, consistency, and official recognition inside the canonical series

This creates a real loop:

- participation earns tokens
- optional actions spend tokens
- a small fraction leaves circulation
- reputation accumulates as a record of contribution rather than as a tradable balance

## Three Candidate Mechanics

These are examples only. They should not be marketed until built.

### 1. Hat Arena

Users submit entries into a themed bracket or faceoff.

- faucet: winners, finalists, or featured entries receive token rewards
- sink: optional premium entry lane or rematch entry uses tokens
- reputation layer: placements and repeated appearances build arena standing for the canonical series

### 2. Spotlight Board

Users compete for a monthly featured placement or community board slot.

- faucet: featured creators receive token payouts from the monthly spotlight budget
- sink: optional cosmetic boosts, badge claims, or naming rights use tokens
- reputation layer: recurring features raise creator standing, credibility, and canon visibility

### 3. Monkey Trials

Users join timed challenges, riddles, remix chains, or event quests.

- faucet: completion rewards or leaderboard payouts distribute tokens
- sink: optional challenge tickets or bonus-attempt entries use tokens
- reputation layer: completed trials and streaks build long-term series reputation

## Canonical Series Rule

If the project introduces a canonical game series, the audience should be able to understand the rule immediately:

- tokens let you participate, signal, enter, upgrade, or stake attention inside the environment
- reputation decides who has actually built standing inside that world

That is a much stronger model than simply rewarding whoever holds the most tokens.

## Anti-Gimmick Rules

MonkeyWifHat should not do any of the following:

- tax ordinary transfers
- claim that deflation alone creates value
- add sinks without a real activity
- market a faucet that has no budget cap or criteria
- pretend a future loop already exists when it does not

## Public Explanation

If MonkeyWifHat later adds a game loop, the explanation should stay simple:

The token stays fixed-supply and transfer-friendly. People can earn tokens through visible community activity and later use tokens inside optional game environments, entries, cosmetics, or canon-linked events. A small part of that optional spend can leave circulation. Reputation inside the canonical series works more like karma: it is earned through participation and recognized history, not simply bought with token balance.
